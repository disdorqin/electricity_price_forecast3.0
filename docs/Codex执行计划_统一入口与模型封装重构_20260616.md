## 2. 已确认的新范围

### 2.1 不再按旧方案机械推进

此前通用重构方案中，默认所有模型都要统一拆成：

- `dataprocess.py`
- `model.py`
- `pipeline.py`

现在根据用户最新要求，范围已调整，不再对所有模型一刀切。

### 2.2 本轮真实约束

1. `lightGBM` 不拆分目录结构，只在统一流水线中接入其现有路径和脚本。
2. `TimesFM` 不拆分目录结构，只在统一流水线中接入其现有路径和脚本。
3. `SGDFNet`、`RT916_SpikeFusionNet`、`TimeMixer` 继续按文档要求整理，但以最小改动、结果一致优先。
4. 统一入口参考 `epf` 项目的组织方式，但不能破坏本项目现有正式融合体系。
5. 融合学习器不重写，直接复用项目里已经做好的正式程序和文档约定。
6. 所有工作只在 D 盘真实项目中执行，不再在 C 盘工作树中做实验性实现。


## 3. 关键背景结论

### 3.1 正式融合路径已经存在

从以下文档和程序确认：

- `docs/PROJECT_ENTRYPOINTS.md`
- `docs/FUSION_V1_STATUS.md`
- `fusion/README.md`
- `fusion/run_full_fusion_suite.py`
- `fusion/pipeline_common.py`

当前项目的正式融合路径已经明确，不应重新设计一套新的融合训练逻辑。

### 3.2 融合的正式思路

当前正式流程是：

1. 各基模型先生成验证期预测表
2. 将验证期预测与真实值汇总
3. 按 `dayahead` / `realtime` 分开拟合
4. 按 `1_8` / `9_16` / `17_24` 分段拟合
5. 学到固定分段权重
6. 将权重应用到正式预测窗口

补充说明：

- 当前正式 V1 路径不是默认 ridge meta learner
- 正式 V1 是“固定分段权重”路径
- TimeMixer 当前 fusion-facing 默认 candidate 已被冻结并在文档中锁定

### 3.3 官方模型池

根据 `fusion/README.md` 和 `docs/FUSION_V1_STATUS.md`：

- `dayahead`：`LightGBM`、`TimesFM`、`TimeMixer`
- `realtime`：`RT916_SpikeFusionNet`、`SGDFNet`、`TimesFM`、`TimeMixer`


## 4. 本轮执行总目标

### 4.1 泄漏修复

修复两处明确的数据泄漏：

1. `SGDFNet`
   - `delta.shift(1)` 类 intraday 特征使用了当日 RT
   - 必须改为 cutoff-safe
2. `RT916_SpikeFusionNet`
   - history 窗口使用了 `actual` 列
   - 应改为 `forecast` 列
   - ramp 相关特征也同步改为 forecast 侧

### 4.2 模型接入整理

1. `lightGBM`：保留原结构，只做统一入口接入
2. `TimesFM`：保留原结构，只做统一入口接入
3. `SGDFNet`：整理成稳定的外部 pipeline 接口，并归档无关产物
4. `RT916_SpikeFusionNet`：整理成稳定的外部 pipeline 接口，并归档无关产物
5. `TimeMixer`：围绕当前正式增强路径做最小重构，整理正式运行所需文件与归档目录

### 4.3 统一入口

创建或收敛根目录统一入口，支持：

- `--pipeline train`
- `--pipeline predict`
- `--pipeline evaluate`
- `--pipeline fusion`

以及：

- `--target dayahead/realtime/both`
- `--models all` 或 指定模型列表

### 4.4 并行执行

不同模型并行执行，至少实现：

- CPU 模型池：`LightGBM`、`SGDFNet`
- GPU 模型池：`TimeMixer`、`TimesFM`、`RT916`

保底策略：

- GPU 模型串行
- CPU 模型并行


## 5. 执行顺序

### 阶段 1：泄漏修复复核与补齐

#### 5.1 SGDFNet

目标：

- 修复 delta 特征和相关 rolling 特征的 intraday 泄漏

原则：

- 只能使用 `D-1 15:00` 时点之前已知的数据

执行方式：

1. 复核 `data_contract.py` 中 delta 家族特征
2. 检查 rolling / 同小时统计 / 局部统计是否仍间接使用当日 RT
3. 保证训练与推理走同一套 cutoff-safe 特征函数
4. 如完全修复复杂度过高，则执行文档保底策略：
   - `delta_lag_1 -> delta_lag_24`
   - 训练和推理共用同一安全实现

#### 5.2 RT916_SpikeFusionNet

目标：

- history encoder 相关输入不再读取 `actual` 列

执行方式：

1. 复核 `dataprocess.py`
2. 复核 `core.py` 或实际拼装 history feature 的调用链
3. 将 history / ramp / gap 类特征统一切到 forecast 侧
4. 检查是否仍存在旁路 actual 引用

#### 5.3 阶段 1 自检

每完成一轮修复，按 autonomous-research-loop 的核心节奏自检：

- `ASSESS`：当前泄漏是否已闭合
- `DECIDE`：继续深修还是采用最小安全修复
- `ESCAPE` 条件：连续 2 次改动改善过小或复杂度失控


## 6. 阶段 2：模型侧整理

### 6.1 lightGBM

本轮策略：

- 不拆目录
- 不改核心 train/infer 脚本组织
- 只做统一调度层接入

预期产物：

- 一个稳定 runner / adapter
- 能从统一入口触发现有脚本

### 6.2 TimesFM

本轮策略：

- 不拆目录
- 不改 `src/timesfm/`
- 不强行重命名主脚本
- 只做统一调度层接入

预期产物：

- 一个稳定 runner / adapter
- 统一入口可以调用现有 TimesFM 预测逻辑

### 6.3 SGDFNet

本轮策略：

- 保留现有 `src/sgdfnet/` 主体
- 通过包装层暴露标准化外部接口
- 归档废弃 config、输出和过程文档到 `_archive/`

重点：

- 不追求推翻现有结构
- 重点是形成统一可调的对外入口

### 6.4 RT916_SpikeFusionNet

本轮策略：

- 在现有主体基础上补稳定 pipeline 包装层
- 将旧说明、旧脚本归档到 `_archive/`

重点：

- 保留当前可运行主链
- 避免因结构整理影响结果

### 6.5 TimeMixer

本轮策略：

- 围绕当前正式增强路径做最小重构
- 不能破坏 fusion 当前默认 candidate 和正式 runner

重点依赖：

- `TimeMixer/enhanced_pipeline.py`
- `fusion/runners/run_timemixer_enhanced_export.py`
- `TimeMixer/candidate_configs/module_b_spike_residual_v1.json`

归档原则：

- 历史输出和实验产物移入 `_archive/`
- 正式运行所依赖的文件必须保留在主路径

## 8. 阶段 4：并行执行策略

### 8.1 基本策略

- CPU 模型并行执行
- GPU 模型默认串行执行

### 8.2 计划实现

统一入口层使用：

- `ProcessPoolExecutor`

但模型实际运行尽量保持：

- 独立 runner
- 独立子进程

原因：

- 避免模型 import 相互污染
- 降低 CUDA 上下文冲突概率
- 降低 GPU OOM 风险

## 10. 已知风险与处理规则

### 10.1 泄漏修复过深导致结果大幅变化

这是预期风险，不视为失败。

要求：

- 记录修复前后差异
- 区分“结果变差”与“结果变真实”

### 10.2 模型整理影响结果一致性

处理原则：

- 先做最小改动整理
- 先验证行为不变
- 再考虑进一步美化结构

### 10.3 GPU 资源不足

处理原则：

- GPU 模型串行
- CPU 模型并行

### 10.4 不确定特征是否泄漏

统一判断原则：

- 如果在 `D-1 15:00` 时点不能知道，就不能用
