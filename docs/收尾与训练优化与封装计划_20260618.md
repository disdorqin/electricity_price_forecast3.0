## 二、训练时间优化（不降低预测精度）

### 2.1 当前耗时分布

| 模型 | 设备 | 训练方式 | 单次验证阶段调用次数 | 预估单次耗时 | 总耗时 |
|------|------|---------|---------------------|------------|--------|
| LightGBM DA | CPU | train-once | 1 | ~6秒 | ~6秒 |
| LightGBM RT | CPU | **逐日重训** | 30 | ~6秒/天 | ~3分钟 |
| TimeMixer | GPU | train-once | 1（但内部 6 子模型） | 2-5分钟 | 2-5分钟 |
| SGDFNet | CPU | **逐日重训** | 30（或 120 含分段） | 2-6秒/天 | 1-6分钟 |
| RT916 | GPU | train-once | 1（6 子模型） | 6-16分钟 | 6-16分钟 |
| TimesFM | GPU | 零样本 | 30（逐日推理） | 1-3秒/天 | 30-90秒 |

**总计估算：15-35分钟/目标（DA 或 RT），两者串行的话 30-70分钟。**

### 2.2 优化方案

#### 优化 1：LightGBM RT 改为 train-once（预估节省 2.5 分钟）

**当前问题**：`run_precision_simulation()` 对每一天重新训练 4 个 LightGBM 模型，但训练窗口只滑动 24 行，模型变化极小。

**方案**：仿照 `run_precision_simulation_da()` 的模式，在循环外训练一次，循环内只做预测。

**风险**：RT 有 D-1 15:00 cutoff 协议（不能用 D 日 RT 实际价格），但训练截止日可以统一为 run_date - 1 的 15:00，对所有验证日和预测日都满足 cutoff。

**修改文件**：`lightGBM/main_fix.py` 的 `run_precision_simulation()` 函数

#### 优化 2：SGDFNet 启用 early_stopping + 改为周期重训（预估节省 80%）

**当前问题**：`HistGradientBoostingRegressor` 的 `early_stopping=False` 被硬编码，且逐日全量重训。

**方案**：
- 在 `models.py` 中将 `early_stopping` 改为 `True`，设 `n_iter_no_change=10`
- 在 `protocol_b_cutoff.py` 中将逐日训练改为每周重训一次（每 7 天），中间日用上一次训练的模型推理
- 可选：启用 `warm_start=True` 做增量训练

**修改文件**：
- `SGDFNet/src/sgdfnet/models.py`（HGBModelConfig）
- `SGDFNet/src/sgdfnet/protocol_b_cutoff.py`（decision_days 循环逻辑）

#### 优化 3：TimesFM 批量推理（预估节省 50% 推理时间）

**当前问题**：`predict_price_for_range()` 逐日调用 `predict_price_for_date()`，每次都重新加载/初始化模型。

**方案**：将模型加载提取到循环外，一次加载后批量推理所有日期。

**修改文件**：`TimesFM/infer.py` 的 `predict_price_for_range()` 函数

#### 优化 4：数据预加载与缓存（预估节省 30-60秒/模型）

**当前问题**：RT916 和 LightGBM 每次都调用 `pd.read_excel()` 读取主数据文件（~39000行），且重复计算特征工程。

**方案**：
- 在 `staged_pipeline.py` 的 `run_model_stage()` 中统一加载一次数据
- 将 DataFrame 传给各模型 pipeline（而非传文件路径）
- 或者：将 xlsx 转为 parquet/csv 缓存，后续读缓存

**修改文件**：
- `pipelines/staged_pipeline.py`（数据预加载）
- 各模型 pipeline.py（支持接收 DataFrame）

**注意**：此优化改动较大，可放到封装阶段一起做。

#### 优化 5：TimeMixer/RT916 模型缓存（预估节省 50% 第二次调用时间）

**当前问题**：model_stage 对每个模型调用两次（验证 + 预测），每次都重新训练。

**方案**：对 train-once 模型（TimeMixer、RT916），在验证调用后序列化模型权重到临时文件，预测调用时加载权重而非重新训练。

**修改文件**：`TimeMixer/pipeline.py`、`RT916_SpikeFusionNet/pipeline.py`

**注意**：此优化需要修改 pipeline 接口，改动较大，ROI 不高，建议放后。

### 2.3 优化优先级

| 优先级 | 优化项 | 预估节省 | 改动量 | 风险 |
|--------|--------|---------|--------|------|
| P0 | LightGBM RT → train-once | ~2.5分钟 | 小 | 低 |
| P1 | SGDFNet early_stopping + 周期重训 | 1-5分钟 | 中 | 中 |
| P2 | TimesFM 批量推理 | 30-45秒 | 小 | 低 |
| P3 | 数据预加载 | 30-60秒/模型 | 大 | 中 |
| P4 | 模型缓存 | 视情况 | 大 | 中 |

建议只做 P0 + P1 + P2，总节省约 4-8 分钟，改动可控。

## 三、项目封装（对标 epf 项目）

### 3.1 当前项目 vs epf 项目对比

| 维度 | epf 项目 | 当前 elec 项目 | 需要改进 |
|------|---------|---------------|---------|
| 入口 | `main.py` + `--pipeline` | 已有 `main.py` + `--pipeline`（四阶段） | 统一为 epf 风格的 predict/train/evaluate |
| 配置 | YAML 默认 + dataclass 校验 + CLI 覆盖 | argparse only | 加 configs/ YAML + dataclass |
| 模型隔离 | subprocess per model | in-process 调用 | 可选改为 subprocess |
| 输出格式 | 统一 `["时刻", "预测值"]` CSV | 已统一 | ✓ 已达标 |
| 工具层 | `utils/` 共享 | 已有 `utils/` | 补充 logging_config |
| 并行 | subprocess 串行 | 串行 | 可加 ProcessPoolExecutor |
| 文档 | README + 陪跑步骤 | 无 README | 需补 |

### 3.2 封装目标

不做大规模重构（风险太高），而是做"最小封装"使项目规范可用：

1. **补 configs/**：为四阶段各建一个 YAML 默认配置
2. **补 README.md**：项目说明 + 快速启动指南
3. **补 requirements.txt**：确保完整
4. **清理 main.py**：统一 CLI 风格，补充 predict/train/evaluate 别名映射到四阶段
5. **统一 runner 子进程入口**（可选）：仿 epf 加 `runners/run_{model}.py`
6. **补 陪跑步骤.md**：运维操作手册
