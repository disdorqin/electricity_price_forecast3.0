## 3. 指标口径

统一按 `docs\metrics_calculation.md` 执行。

当前必须遵守的核心口径：

- `SMAPE` 计算前对真实值与预测值执行 `max(value, 50)`。
- `SMAPE` 结果使用百分比口径。
- `MAE / MSE / R2` 按常规定义。
- `SCR / 度电套利 / 改良度电套利` 在单模型结果稳定后统一补齐。

- 进入正式对标表之前，必须补齐完整业务指标。

## 4. 无泄露约束

### 4.1 基本原则

- 预测某一天时，只能使用该预测决策时点之前可见的数据。
- 不允许把未来真实值提前注入训练、验证或推理输入。

### 4.2 日滚动预测逻辑

模型按天输出未来预测时，预测第 `D+1` 天后，预测第 `D+2` 天时：

- 可以使用第 `D+1` 天已经发生的真实值作为历史输入回填。
- 这属于真实部署场景中的合法信息更新，不属于泄露。

进一步约束：

- 日前任务预测某日时，不能提前看到该日真实日前价格。
- 实时任务预测某日时，必须严格卡在 `D-1` 日 `15:00` 的信息截面。
- 如果模型支持 DA 注入 RT，则注入的只能是该时点可得的 DA 信号或外部预测 DA，不能注入未来 RT 真值。

### 4.3 需要重点核查

需要对以下脚本逻辑做核查，确认它们符合上面的时序约束：

- `D:\作业\science\大创科研时序\代码\elec\scripts\runners`
- `electricity_forecast_model2.0\fusion\runners`
- 各模型自己的 daily backtest / joint backtest 入口

当前已确认的入口事实：

- `run_marketlinked_daily_asof.py` 默认带 `--asof-hour 15`
- `run_spiketimesnet_daily_asof.py` 默认带 `--asof-hour 15`
- `SpikeTimesNet.core.run_daily_asof_backtest(...)` 与 `MarketLinkedSpikeTimesNet.core.run_daily_asof_backtest(...)` 都是按 `D-1 15:00 -> 预测 D 日` 的日滚动接口设计
- `SpikeTimesNet.core.apply_asof_cutoff_for_inference(...)` 与 `MarketLinkedSpikeTimesNet.core.apply_asof_cutoff_for_inference(...)` 都会在推理前对 `asof` 之后的实时电价做物理截断
- `SpikeTimesNet.core.run_joint_da_rt_daily_backtest(...)` 与 `MarketLinkedSpikeTimesNet.core.run_joint_da_rt_daily_backtest(...)` 都是先跑 DA，再把当日可得的 DA 预测注入 RT
- 两个核心文件目前都把训练窗口默认写成 `months=12`，因此单模型复现阶段应先沿用这个协议，不要再私自缩到 6 个月

核查重点：

1. 预测次日时是否只使用了前一日已知真实值。
2. 实时任务是否严格卡在当日 `15:00` 截止信息。
3. 日前任务是否严格卡在日前决策时点。
4. 缺失值填充是否错误地用了未来真实值。

## 5. 计算资源约定

### 5.1 环境

- GPU 相关任务统一使用 `conda epf-2`
- CPU 任务允许直接用当前 Python 环境运行

### 5.2 并行原则

- `LightGBM` 优先走 CPU
- `TimeMixer / TimesFM / RT916` 优先走 GPU
- 同一时刻避免多个重 GPU 任务并行争抢显存
- 可以采用“CPU + 1 个 GPU 任务”并行

执行口径：

- 所有正式复现命令默认在 `conda epf-2` 下执行。
- 若某个脚本天然只吃 CPU，可在同一时段与一个 GPU 模型并跑。
- 除非显存验证通过，否则不允许 `TimeMixer / TimesFM / RT916` 三者两两同时抢同一张 GPU。

### 5.3 当前建议

- `LightGBM` 与 `TimeMixer` 可分配到 CPU/GPU 同时跑
- `TimesFM` 与 `RT916` 更适合串行占用 GPU

## 6. 复现协议

### 6.1 总体协议

采用“按月重训、按月预测、最后拼接”的协议。

原因：

- 历史表更接近日滚动训练，但我们没必要强行完全复现到日训成本。
- 整段一次训练偏差过大，不足以作为复现结论。
- 按月重训是当前时间成本和效果之间更合理的折中。

### 6.2 月度协议

对于某个月份 `M`：

1. 使用 `M` 之前的固定历史窗训练。
2. 在训练窗内部保留验证集。
3. 训练完成后，对该整个月份做逐日预测。
4. 预测下一天时，允许把前一天的真实值回填到历史输入。
5. 月末输出整月结果。

推荐的复现粒度：

- 先以“整月重训一次，整月逐日推理”作为基础协议。
- 如果某个模型整月协议仍明显偏离历史表，再进一步尝试“半月重训”或“更短更新周期”，但这属于第二轮优化，不作为第一轮默认协议。

### 6.3 训练窗

初始默认：

- 历史窗从 `12` 个月开始
- 训练内验证先按 `8:2`

说明：

- 如果某个模型在 `12` 个月下仍明显偏离历史表，可调整训练窗长度。
- 调整优先级：
  1. 训练时间段
  2. 训练/验证切法
  3. 模型参数

当前不优先改模型参数，除非训练协议收齐后仍无法接近历史表。

## 7. 执行顺序

### 7.1 第一阶段：先测 2026 年 2 月

先选择 `2026-02-01 ~ 2026-02-28` 作为首轮主对标月份。

原因：

- 当前已经对 2 月做过多次调试，最方便快速验证。
- 可以先看协议是否正确，再扩展到其他月份。
- 如果 2 月都偏差很大，说明不是月份偶然性，而是复现链本身有问题。

补充原因：

- 历史总表里 `TimeMixer` 的日前/实时覆盖从 `2026-02-24` 开始，2 月是排查它与其他模型协议差异的最低成本切入口。
- 2 月结果稳定后，再扩到 3 月、4 月，可以更快分辨“协议错误”与“月份偶然性”。

第一阶段的具体执行顺序固定为：

1. 先做单模型复现，不做融合。
2. 先跑 `TimeMixer / TimesFM / LightGBM / RT916` 各自的 2026 年 2 月单月结果。
3. 单模型达到历史表邻近水平后，再扩 3 月、4 月。
4. 只有单模型稳定后，才允许进入学习器或融合器调优。

### 7.2 第二阶段：扩到 3 月、4 月

当 2 月达到可接受复现水平后：

- 扩到 `2026-03`
- 再扩到 `2026-04`

目的：

- 检查模型是否只是在 2 月碰巧有效
- 验证复现协议是否跨月稳定

### 7.3 第三阶段：再回到 1-5 月整段

在 2 月、3 月、4 月都达标后，再拼 `2026-01 ~ 2026-05`。

## 10. 判断是否达标

复现达标不要求逐点完全一致，但要求：

1. 整体趋势与历史表一致
2. 主力模型排序基本一致
3. `SMAPE` 与历史表差距处于可接受范围

当前判断标准建议：

- 如果某模型比历史表差 `1~3` 个点，属于可继续优化范围
- 如果差距超过 `5` 个点，优先判断为协议或脚本问题
- 如果分时段某一段异常偏离，优先查该段输入和覆盖

当前单模型复现的目标不是“逐点完全相同”，而是满足下面三条：

1. 模型排序大致和历史汇总表一致。
2. 核心月份 `2026-02 / 2026-03 / 2026-04` 的整体 SMAPE 不再出现 `40+ / 50+` 这类明显失真结果。
3. 同一模型不同月份的分时段表现要有业务可解释性，不能只靠某一段偶然拉低均值。

- 当前 `fusion/runners/run_lightgbm_export.py` 实际导入的是 `electricity_forecast_model2.0\lightGBM\main_fix.py`，不是 `epf` 原始动态寻优版
- 因此后续判断 LightGBM 是否复现达标时，要明确说明“这是当前固定窗复现版结果”，不能和 `epf` 原始动态寻优版混为一谈

### 11.4 TimesFM

- 当前主要问题不是能不能跑，而是是否真的按历史协议产出了同等级效果
- 因为缺少完整历史逐点明细，优先先对齐历史汇总表水平

### 11.5 数据快照差异

## 12. 后续进入融合前的前提

只有在以下条件满足后，才进入学习器阶段：

1. 前日模型池单模型已基本复现
2. 实时模型池单模型已基本复现
3. 各模型输出表结构统一稳定
4. 指标计算口径统一

满足后再做：

- 从验证期导出 `task / period / hour_business / y_true / y_pred / model_name`
- 训练轻量权重学习器
- 最终生成固定权重或轻量 meta-learner

## 12.1 学习器既定设计

当前学习器设计先固定为“验证期导出预测结果，再学习分时段固定权重”，不直接改动各单模型内部训练逻辑。

统一思路：

1. 每个单模型先按各自复现协议完成一次训练。
2. 训练窗内部统一按 `8:2` 做时间顺序划分：
   - 前 `80%` 作为训练集
   - 后 `20%` 作为验证集
3. 单模型在验证集上输出预测结果表。
4. 将同一任务下所有模型的验证期结果汇总后，交给学习器学习权重。
5. 学习器学到的是固定权重，不带截距，不额外引入 `pred_mean`、`hour_business`、`intercept` 之类附加项。
6. 学到的权重按任务、按分时段固定下来，再用于后续真实预测区间的加权融合。

验证期导出表结构统一为：

- `task`
- `period`
- `hour_business`
- `y_true`
- `y_pred`
- `model_name`

任务侧处理方式：

- 日前模型验证结果先单独汇总，训练日前学习器。
- 实时模型验证结果先单独汇总，训练实时学习器。
- 日前和实时不混在同一个学习器里。

分时段处理方式：

- `1_8`
- `9_16`
- `17_24`

三个时段分别学习各自权重。

最终使用方式：

1. 先完成单模型真实预测。
2. 再读取该任务、该时段对应的固定权重。
3. 对单模型预测结果做加权。
4. 输出融合后的最终预测表。

当前阶段默认约束：

- 学习器只使用验证期预测结果，不直接用测试期或真实预测期信息。
- 学习器只做后融合，不反向修改单模型参数。
- 如果单模型复现不达标，则不推进学习器训练。

## 17. 当前最重要的判断标准

如果单模型复现不达标，不要急着继续优化学习器。

原因很直接：

- 融合器只能放大已有单模型能力，不能替代单模型本体训练质量。
- 如果单模型当前版本比历史版本弱很多，学习器学到的只会是“在弱模型之间做权衡”，无法代表最终上线能力。
- 因此当前最优先目标仍然是：先让单模型复现水平尽量靠近 `模型结果汇总.xlsx`。
